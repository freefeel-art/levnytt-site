# LevNytt Morning Brief Architecture

**Note:** This document describes the Morning Brief subsystem. For the complete agent inventory, see `docs/specifications/AGENT-REGISTRY.md`.

## Overview

The LevNytt Morning Brief system consists of two independent intelligence agents that run every morning. Both analyze different information sources, produce independent recommendations, and return control to the editor. The editor compares both briefs and decides the single highest-value action for the day.

| Agent | File | Purpose |
|---|---|---|
| Morning Brief | `.opencode/agents/morning-brief.md` | Consumer Intelligence |
| MLM Brief | `.opencode/agents/mlm-brief.md` | Industry & Editorial Intelligence |

Neither agent publishes content. Neither writes articles. Both only make editorial recommendations.

The Morning Brief system is part of a larger editorial agent ecosystem documented in the canonical Agent Registry.

---

## Mission

The Morning Brief system is the daily editorial decision apparatus for LevNytt.

Its purpose is to analyze both external consumer demand and internal editorial readiness, then surface the single highest-value action for the day.

The system never publishes content.

It never writes articles.

It only makes editorial recommendations.

---

## Position in the Workflow

```
  Morning Brief            MLM Brief
  (consumer intel)    (industry intel)
         ↓                    ↓
     ───────── Human Approval ─────────
                    ↓
            Publication Agent
                    ↓
            Published Content
```

Both briefs are executed first every morning. Control always returns to the user. No other agent is executed automatically. Human approval is always required before continuing.

## Agent Responsibilities

### Morning Brief — Consumer Intelligence

The Morning Brief analyzes what consumers are searching for and discussing today.

**Inputs:**

External signals only:
* Google Trends
* Reddit discussions (r/sweden, r/Supplements, r/Biohackers)
* Google Autocomplete
* Question-intent Autocomplete proxy (in lieu of People Also Ask)
* Published content (for coverage comparison)
* Sitemap

**Responsibilities:**
* Detect what consumers are searching for today
* Detect trending topics and emerging consumer interests
* Compare external demand with existing published coverage
* Identify content gaps from the consumer perspective
* Detect new consumer language and framing
* Recommend one primary consumer-facing content opportunity
* Recommend one secondary opportunity if appropriate

**Does NOT analyze:**
* Editorial backlog (gap reports)
* Authority research status
* Production status
* Editorial briefs history
* MLM/regulatory pipeline

### MLM Brief — Industry & Editorial Intelligence

The MLM Brief analyzes the internal editorial pipeline and industry landscape.

**Inputs:**

Internal signals:
* Editorial Backlog (gap reports)
* Editorial Briefs
* Production Status
* Authority Research
* Existing Articles
* Sitemap
* Published HTML pages
* Industry context (MLM, direct selling, FTC, regulatory)

**Responsibilities:**
* Analyze authority research coverage and gaps
* Track editorial backlog state
* Monitor production status (generated vs published)
* Detect coverage gaps from editorial backlog perspective
* Detect outdated priorities and stale gap reports
* Identify production blockers
* Identify missing authority research
* Recommend one primary editorial/pipeline action
* Recommend one secondary action if appropriate

**Does NOT analyze:**
* Google Trends
* Reddit discussions
* Google Autocomplete
* Consumer search behavior
* External market signals

## Integration with Editorial Sync Engine

As of Sprint 17, all derived editorial state is generated by the Editorial Sync Engine (`docs/specifications/EDITORIAL-SYNC-ENGINE.md`). The MLM Brief no longer performs cold-scan inventory, drift detection, or production queue computation inline.

### Pre-execution Flow

```
Before MLM Brief:
  1. Sync Engine runs (or has run since last state change)
  2. Generated files are current:
     - docs/editorial-backlog/production-status.md
     - docs/reports/repository-inventory.md
     - docs/reports/drift-detection.md
     - docs/reports/publication-queue.md
  3. MLM Brief reads generated files
  4. MLM Brief performs analysis requiring AI judgment only
```

### Agent Responsibilities (Updated)

**MLM Brief** now reads Sync Engine generated files instead of computing its own inventory. Its remaining human-judgment responsibilities:
- Analyze authority research coverage against publication queue
- Track editorial backlog state from generated queue
- Assess strategic trade-offs between queue items
- Recommend primary pipeline action based on Sync Engine state
- Apply industry context (MLM, direct selling, FTC, regulatory)

### Generated Files the MLM Brief Consumes

| Generated File | What It Replaces (inline in brief) |
|----------------|-----------------------------------|
| `production-status.md` | Manual per-article status tracking |
| `repository-inventory.md` | Manual page count + categorization |
| `drift-detection.md` | Manual drift findings section |
| `publication-queue.md` | Manual ranked queue construction |

## Editorial State Interface

Both Morning Brief and MLM Brief agents consume synchronized editorial state generated by the Editorial Sync Engine. This eliminates redundant repository analysis and ensures consistent data across agents.

**Interface Specification:** `docs/editorial-state/README.md`

**Key Architecture Principle:** Never inspect the repository directly if synchronized editorial state already exists. Repository traversal occurs only within the Editorial Sync Engine.

**Synchronized Files Consumed:**
- `docs/editorial-backlog/production-status.md` (MLM Brief)
- `docs/reports/publication-queue.md` (MLM Brief)  
- `docs/reports/repository-inventory.md` (Both agents)
- `docs/reports/drift-detection.md` (Future agents)

## Shared Guiding Principle

Morning Brief is not an AI writer.

MLM Brief is not an AI writer.

Both are editorial decision engines.

Their shared success is measured by helping the editor consistently choose the highest-value work each day — whether driven by consumer demand or editorial pipeline readiness.
