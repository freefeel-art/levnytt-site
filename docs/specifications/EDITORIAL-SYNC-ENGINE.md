# Editorial Sync Engine — Specification V1.0

**Status:** Specification — Sprint 17
**Date:** 2026-06-30
**Design principle:** Repository state is always the single source of truth. All editorial tracking is derived from canonical sources, never duplicated.

---

## 1. Overview

The Editorial Sync Engine is a deterministic process that scans canonical repository sources and derives the complete editorial state. It eliminates duplicated tracking, stale backlog recommendations, and manual state maintenance.

The Sync Engine does not write articles. It does not publish content. It only reads canonical sources and generates derived state files.

---

## 2. Canonical Sources of Truth

These are the ONLY sources the Sync Engine reads. No tracking file is used as a source — only what physically exists in the repository.

| Source | Path | What It Provides |
|--------|------|-----------------|
| Published pages | `/*.html` (root) | Every published HTML page. Canonical for "what is live." |
| Source articles | `content/articles/*.html` | Every completed article not yet at root. Canonical for "what is ready to publish." |
| Gap reports | `docs/editorial-backlog/*-gap.md` | Content gap analysis — human editorial research. Each gap report is a cluster of content opportunities with priorities, scores, and strategic fit. |
| Authority research | `docs/authority-research/*.md` | Completed research documents. A file's existence means a topic domain is cleared for article production. |
| Product entities | `content/products/entities/*/sv.json` | Structured product metadata for all NeoLife products. Canonical for product identity. |
| Price database | `docs/databank/LEVNYTT-PRICE-DATABASE.md` | Current pricing for all products. |
| Brand assets | `assets/brand/*` | Required brand files. Canonical for brand readiness. |
| Product entity categories | `content/products/categories.json` | Controlled vocabulary for product classification. |

### NOT canonical (not used as Sync Engine inputs)

| Source | Reason |
|--------|--------|
| `production-status.md` | Derived file — will be generated by the Sync Engine. |
| Editorial briefs | Derived files — generated by agents reading Sync Engine output. |
| `docs/reports/*` | Derived files — generated by audit/analysis. |
| `sitemap.xml` | Derived — generated from root pages. |
| `docs/specifications/*` | Design documents — not repository state. |
| `.opencode/agents/*` | Agent definitions — not repository data. |

## 3. Derived Outputs (Auto-Generated)

The Sync Engine generates these files. Human authors never edit them directly.

### 3.1 `docs/editorial-backlog/production-status.md`

**Purpose:** Master production tracker — the single authoritative view of every article's status.

**Content (auto-generated from canonical sources):**

| Column | Source |
|--------|--------|
| Item # | Sequential |
| Cluster | Gap report cluster name |
| Topic | Article topic |
| Score | From gap report |
| Status | Derived: **Published** if `/*.html` exists; **Generated** if only in `content/articles/`; **New Article** if only in gap report; **Update Existing** if gap reports recommends update |
| File | Root filename (published) or source filename (generated) |
| Authority | Derived from `docs/authority-research/` — CLEARED or BLOCKED |
| Gap Report | Source gap report filename |
| Cluster # | Cluster number within gap report |

**Logic:**

```
FOR each gap report in docs/editorial-backlog/*-gap.md:
  FOR each cluster in gap report:
    IF cluster recommends "Skip":
      SKIP
    IF root file exists matching topic → Status = Published
    ELSE IF content/articles/ file exists matching topic → Status = Generated
    ELSE IF cluster recommends "Update existing" → Status = Update Existing
    ELSE → Status = New Article (unwritten)

    Authority = CLEARED if authority-research/ has matching file else BLOCKED
```

### 3.2 `docs/reports/repository-inventory.md` (NEW — replaces SITE-PAGE-INVENTORY.md)

**Purpose:** Comprehensive inventory of all repository editorial assets.

**Content:**

```
Published Pages:
  - Count: N
  - By category: product, pillar, informational, landing, utility
  - By source gap report
  - Orphan pages (no gap report source)

Source Articles (content/articles/):
  - Count: N
  - Published duplicates (root file exists)
  - Unique source-only (served via rewrite)
  - Orphan articles (no gap report source)
  - Articles stale (root file exists without matching source — pipeline bypass)

Product Entities:
  - Count: N
  - By category
  - Entities with product pages
  - Entities without product pages
  - Entities with variants
  - Inactive entities

Authority Research:
  - Count: N
  - By associated gap report (MLM, Omega-3, Kosttillskott)
  - By status (CLEARED / BLOCKED)
  - Total CLEARED topic domains
  - Total BLOCKED Priority A topics

Gap Reports:
  - Count: N
  - By status (exhausted, active, stale)
  - Stale clusters (recommendations that contradict published state)
```

### 3.3 `docs/reports/drift-detection.md` (NEW — replaces inline drift section in editorial briefs)

**Purpose:** Automatically detected inconsistencies between repository state and editorial plans.

**Detects:**

| Drift Type | Detection Logic |
|------------|----------------|
| Stale gap cluster | Gap report recommends "Write now" but root page already exists |
| Undocumented page | Root page exists but no gap report cluster matches |
| Source file → root file present | content/articles/ file also exists at root (duplicate) |
| Source file → no gap report | content/articles/ file with no matching gap report cluster |
| Published → no production-status | Root page not appearing in any production-status entry |
| Authority CLEARED → not in production queue | Research exists but no article planned in any gap report |
| Gap report → stale authority status | Report marked "needs authority research" but research now exists |

### 3.4 `docs/reports/publication-queue.md` (NEW)

**Purpose:** Ranked list of next articles to produce. Generated from gap report priorities × authority status × production status.

**Ranking logic:**

```
1. Priority A, Authority CLEARED, not Published, not Generated
2. Priority A, Authority BLOCKED (shows blocker)
3. Priority B, Authority CLEARED, not Published
4. Update Existing, Authority CLEARED
5. Priority B, Authority BLOCKED
```

**Output format:**

```
# Publication Queue
Generated: YYYY-MM-DD

## Ready to Write (Authority CLEARED)
| Rank | Topic | Priority | Score | Gap Report | Authority File |
|------|-------|----------|-------|------------|----------------|

## Blocked (Needs Authority Research)
| Rank | Topic | Priority | Score | Gap Report | Missing Research |
|------|-------|----------|-------|------------|-----------------|

## Ready to Publish (Source file exists)
| Rank | Topic | Score | Source File | Coverage Score |
|------|-------|-------|-------------|----------------|

## Update Existing
| Rank | Topic | Priority | Existing Article | Action |
|------|-------|----------|-----------------|--------|
```

## 4. Human-Authored Documents (Never Auto-Generated)

| Document | Why It Stays Human-Authored |
|----------|---------------------------|
| Gap reports (`docs/editorial-backlog/*-gap.md`) | Human editorial analysis cannot be automated. The insight, strategic fit assessment, and priority judgment require human domain expertise. |
| Authority research (`docs/authority-research/*.md`) | Scientific research synthesis requires expert judgment. Auto-generated summaries would lack the depth and accuracy needed for EEAT. |
| Agent definitions (`.opencode/agents/*.md`) | Agent behavior and rules are human-designed. |
| Specifications (`docs/specifications/*.md`) | Architectural decisions are human-made. |
| Price database (`docs/databank/*.md`) | Pricing is a business decision verified by humans. |
| Brand system (`docs/BRAND-DESIGN-SYSTEM.md`) | Brand decisions require human creative direction. |
| Product entity JSON files | Content authored by research, validated by script. Entities are machine-readable but human-curated. |
| Article HTML files | All article content is human-authored (by agents under human direction) or human-written — never auto-generated by the Sync Engine. |

## 6. Execution Workflow

### 6.1 Trigger Points

The Sync Engine runs deterministically. It produces the same output from the same repository state. It can be run at any time without side effects.

**Required triggers (must run before):**

| Trigger | Why |
|---------|-----|
| Before MLM Brief generation | MLM Brief needs fresh production-status.md, drift detection, publication queue. |
| After every publication | Publication changes root pages and content/articles/ state. Sync must update derived files. |
| After gap report creation | New gap report adds clusters to the production queue and coverage analysis. |
| After authority research | New research file changes BLOCKED → CLEARED status for topic domains. |
| After product entity changes | Entity count/coverage changes affect repository-inventory.md. |

**Optional triggers (can run but outputs won't change unless canonical sources changed):**

| Trigger | Effect |
|---------|--------|
| Before Morning Brief generation | Morning Brief reads published pages directly — does not depend on Sync Engine. No effect. |
| Before editorial planning session | Fresh state for human review. |
| After article generation | Source article added to content/articles/ → production-status.md updates. |
| On demand | Always safe. |

### 6.2 Execution Order

```
1. Scan canonical sources
   1a. List root /*.html (excluding index.html, 404.html, integritetspolicy.html, template files)
   1b. List content/articles/*.html
   1c. List content/products/entities/*/sv.json
   1d. List docs/authority-research/*.md
   1e. Parse all docs/editorial-backlog/*-gap.md (extract clusters, priorities, scores, recommendations)
   1f. List assets/brand/ (verify required files exist)

2. Derive editorial state
   2a. Match published pages to gap report clusters (by topic keyword matching)
   2b. Match source articles to gap report clusters
   2c. Compute authority status per cluster (CLEARED if matching research file exists)
   2d. Compute production status per cluster (Published / Generated / New Article / Update Existing)
   2e. Identify orphan pages (no gap report match)
   2f. Identify stale clusters (recommendation contradicts published state)
   2g. Identify undocumented source articles
   2h. Compute publication queue (ranked by priority × authority × production status)

3. Generate derived files
   3a. Write docs/editorial-backlog/production-status.md
   3b. Write docs/reports/repository-inventory.md
   3c. Write docs/reports/drift-detection.md
   3d. Write docs/reports/publication-queue.md

4. Return structured state
   4a. Return full state as structured data for agents to consume
```

## 7. Integration with Editorial Commander

### 7.1 Current State

The MLM Brief agent currently performs its own repository analysis inline — counting pages, checking authority research files, computing production status, detecting drifts. This duplicates the Sync Engine's analysis and means every brief starts from a cold scan.

### 7.2 Target State

```
Before MLM Brief runs:
  1. Sync Engine executes
  2. Sync Engine generates production-status.md, repository-inventory.md,
     drift-detection.md, publication-queue.md
  3. MLM Brief reads generated files (no cold scan needed)
  4. MLM Brief performs ONLY the analysis that requires AI judgment:
     - Recommend primary action based on sync state
     - Assess strategic trade-offs
     - Evaluate editorial pipeline decisions
     - Industry context analysis
```

### 7.3 Agent Changes Required

**MLM Brief agent (future update):**
- Remove inline page counting (read `repository-inventory.md`)
- Remove inline drift detection (read `drift-detection.md`)
- Remove inline production queue computation (read `publication-queue.md`)
- Remove inline authority research checking (read `publication-queue.md` status)
- Keep: strategic recommendation, pipeline analysis, industry context, trade-off assessment

**Morning Brief agent:**
- No changes needed — reads external signals and root pages directly

**Publication Agent:**
- No changes needed — validates before publishing, updates production-status.md
- Publication Agent triggers Sync Engine after each publication

## 8. Health Status

The Sync Engine determines repository health based on:

| Condition | Status |
|-----------|--------|
| No stale clusters, no undocumented pages, production queue up to date | **GREEN** |
| Stale clusters exist (gap report recommends writing what's already published) | **AMBER** — flag as drift |
| Undocumented pages exist without gap report source | **AMBER** — flag as drift |
| Production queue has unwritten Priority A items with CLEARED authority | **GREEN** — flag as opportunity |
| Production queue has Priority A items BLOCKED with no authority research | **GREEN** — flag as blocker, not a health issue |
| Production-status.md has wrong status for any article | **RED** — Sync Engine should correct this |

## 10. Validation Against Design Principles

| Principle | How the Sync Engine Satisfies It |
|-----------|---------------------------------|
| Single Source of Truth | All derived state comes from canonical sources. No tracking file is a source. |
| Eliminates duplicated tracking | `production-status.md` becomes generated. Inline inventory/drift/queue in editorial briefs are removed. |
| Prevents stale backlog recommendations | Drift detection automatically flags gap clusters whose "Write now" recommendation contradicts published state. |
| Accurately reflects repository state | Sync Engine reads the actual filesystem — not a secondary tracking file that may be stale. |
| Supports future automation | Generated files provide structured data that any tool or agent can consume. State data structure is machine-readable. |
| Integrates cleanly with Editorial Commander | MLM Brief reads generated files instead of computing inline. No architectural disruption. |
| Does not modify existing systems | Sync Engine is additive. It generates new files and replaces generated ones. It does not touch human-authored content, agents, brand, or published articles. |

## 11. Sprint 18 Recommendation

**Implement the Editorial Sync Engine as a Python script.**

### Files to Create (Sprint 18)

| File | Purpose |
|------|---------|
| `scripts/sync-editorial-state.py` | Main Sync Engine implementation |
| `scripts/sync-editorial-state.sh` | Convenience shell wrapper |

### Implementation Approach

1. **Python 3 script** — reads canonical sources, computes state, writes derived files
2. **Deterministic** — no randomness, no external API calls (except filesystem)
3. **Fast** — completes in <1 second (pure filesystem operations)
4. **Self-contained** — no dependencies beyond Python 3 standard library
5. **Safe to run anytime** — never deletes, only writes to designated generated-file paths

### Implementation Steps

1. Parse root directory — identify published pages by category (product slug pattern `neolife-*`, portal guide suffixes, informational)
2. Parse `content/articles/` — list all source files
3. Parse `content/products/entities/` — count entities, categorize
4. Parse `docs/authority-research/` — list files, map to gap report topics
5. Parse gap reports — extract clusters, priorities, scores, recommendations (structured YAML frontmatter or regex)
6. Match canonical sources to gap clusters — topic keyword matching
7. Compute derived state — production status, drifts, queue, inventory
8. Write output files — production-status.md, repository-inventory.md, drift-detection.md, publication-queue.md

After implementation and verification, update the MLM Brief agent to consume generated files.
